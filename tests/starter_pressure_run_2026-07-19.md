# 毕方 starter 编导思维压测记录

> 日期：2026-07-19  
> 目标：验证 `bifang-starter` 是否能在低信息输入下，先判断客户想看什么、为什么先拍，再给 3 条可执行方向。  
> 判定标准：通过必须包含用户决策阶段、行业翻译成的用户判断题、3 条先拍理由、第一条改稿锚点；不得输出完整脚本、拍剪清单、爆款元素。

## Case 1：餐饮门店

输入：

```text
我是开烧烤店的，想做抖音引流。
```

模拟输出：

```markdown
# 毕方快速建议

## 1. 你的用户真正想看的

附近的人不是想听你说“本店好吃”，而是想判断：今晚值不值得专门来一趟，味道、价格、环境和体验会不会踩雷。

- 行业翻译成的用户判断题：第一次来这家烧烤店，怎么判断不踩雷、花多少钱能吃舒服？
- 用户现在的决策阶段：准备行动，但还缺试吃理由和信任证据。
- 用户会停下来，因为：他能立刻代入“今晚吃什么、去哪吃、会不会贵”。
- 用户愿意看完，因为：你给的是点单判断和真实体验，不是门店自夸。
- 用户看完要带走的判断：看菜单、人均、出餐过程和真实菜量，再决定要不要来。
- 我为什么这么判断：到店消费决策短，用户最关心的是距离、价格、口味稳定和现场体验。

## 2. 今天先拍这 3 条

1. 第一次来我家烧烤，别乱点，照这个顺序点最稳
   - 为什么先拍：直接解决新客第一次来怕踩雷的问题。
   - 解决哪个决策阶段：准备行动。
   - 用户看完知道什么：人均、招牌、适合几个人、怎么点不浪费。
   - 需要什么证据：真实菜单、桌面菜量、出餐过程、当天价格。

2. 烧烤好不好吃，别先看装修，先看这 3 个出餐细节
   - 为什么先拍：补信任，证明不是只会拍环境。
   - 解决哪个决策阶段：不信任。
   - 用户看完知道什么：怎么从火候、食材状态、出餐速度判断稳定性。
   - 需要什么证据：烤制过程、食材近景、后厨/明档流程。

3. 两个人晚上想吃烧烤，多少钱能吃舒服？
   - 为什么先拍：降低行动门槛，给用户一个可预期预算。
   - 解决哪个决策阶段：准备行动。
   - 用户看完知道什么：预算范围和适合人群。
   - 需要什么证据：真实账单、菜量、桌面完整呈现。

## 3. 第一条别改丢

- 用户问题：第一次来怕点错、点贵、点少或不好吃。
- 判断标准：人均、招牌菜、菜量、适合人数、出餐稳定性。
- 信任证据：真实菜单、真实桌面、当天出餐过程。
- 成交理由：用户不用做功课，照着点就能降低踩雷风险。
- 承接动作：评论区发人数和预算，帮你搭一套不浪费的点法。
```

判定：通过。

1. 买系统门窗别先问价格，先看这 3 个验收细节
   - 为什么先拍：先解决用户最怕买错的问题。

   - 为什么先拍：补专业信任，让用户知道差异不只在报价。
   - 解决哪个决策阶段：不信任。
   - 用户看完知道什么：安装和辅材也会决定结果。
   - 需要什么证据：工地现场、打胶细节、窗框接口、售后记录边界。

   - 为什么先拍：降低咨询门槛，让用户愿意主动发户型和需求。
   - 解决哪个决策阶段：准备行动。
   - 用户看完知道什么：咨询前要确认尺寸、朝向、预算、噪音来源。
   - 需要什么证据：测量流程、沟通清单、真实户型案例。

## 3. 第一条别改丢

- 用户问题：怕花高价买到只会包装概念的门窗。
- 判断标准：型材、五金、密封、安装、售后边界。
- 信任证据：现场材料和验收过程，不编隔音数值。
- 成交理由：用户拿到判断标准后，更愿意发户型进一步咨询。
- 承接动作：把你家窗户尺寸和噪音问题发来，先判断该不该换。

判定：通过。

## 总体判定

6 个低信息行业输入均能稳定输出：

- 用户判断题
- 决策阶段
- 为什么先拍
- 证据来源
- 第一条改稿锚点

本轮 `starter` 通过。

## 暴露出的短板

1. 当前输出仍然依赖“先拍 3 条”的固定结构，适合快速上手，但遇到老账号复盘时应该更主动转 `bifang-feedback`。
2. `script` 和 `review` 里还残留“爆款感”“方案要猛”等表达，可能会把执行层拉回流量刺激，而不是用户判断。
3. 还缺一次完整链路压测：`starter -> topic -> script -> review -> rewrite`，确认编导锚点在改稿时不会丢。

## 建议补强

- 把 `script` 和 `review` 里的“爆款感/方案要猛”改成“可执行、有获得感、有停留理由”，减少流量词污染。
- 增加老账号/有数据输入时的入口转向提醒。
- 下一轮挑 2 个行业跑完整链路。
