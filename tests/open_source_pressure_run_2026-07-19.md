# 毕方开源版压测记录

> 日期：2026-07-19  
> 目标：验证不依赖私有达人蒸馏资料时，毕方开源基础能力是否仍然成立。  
> 原则：开源版只保留通用编导判断框架，不发布达人研究、课程蒸馏、客户私有数据。

## 1. 开源边界检查

开源版应包含：

- 通用 skill：`starter/topic/script/review/rewrite/feedback/intake/diagnosis/profile/assets/report`
- 通用模板：`templates/starter_output_template.md`
- 通用测试：烟测、行业正向样例、失败回归、开源验收、编导思维压测
- README / CHANGELOG / RELEASE_NOTES

开源版不应包含：

- 私有达人/竞品研究文件
- 私有蒸馏记录文件
- 任何具体达人名单、账号 ID、课程蒸馏、私有客户数据
- 早期工具脚本里带具体达人视角的生成逻辑
- `bifang-v4-orchestrator/` 及其 references；V0.1 开源版先做 skills-only，编排器单独清理后再考虑发布
- `outputs/`、`iterations/`、`sales/`

## 2. 通用低信息压测

### Case A：本地门店

输入：

```text
我开社区面馆，想拍抖音引流。
```

通过标准：

- 能判断用户不是想看老板介绍，而是判断今天值不值得来吃一碗。
- 能给 3 条先拍方向，并覆盖信任、比较、行动。
- 不需要任何达人私有知识。

判定：通过。

### Case B：高客单服务

输入：

```text
我做企业增长咨询，想收线索。
```

通过标准：

- 能判断用户想知道咨询师能不能先判断增长卡点，而不是只讲理论。
- 不承诺结果，不编案例。
- 能给适配边界和低压承接。

判定：通过。

### Case C：课程/IP

输入：

```text
我是职场沟通课老师，想卖课。
```

通过标准：

- 能落到具体场景：汇报、争取资源、被质疑。
- 不承诺升职加薪。
- 第一条优先给用户一个可试用的小判断标准。

判定：通过。

## 3. 完整链路压测

使用已完成的完整链路样例：

- 系统门窗获客
- 职场沟通课卖课

链路：

```text
starter -> topic -> script -> review -> rewrite
```

通过标准：

- 用户判断题不丢。
- 判断标准不丢。
- 信任证据不丢。
- 成交理由不丢。
- 承接动作不丢。

判定：通过。

## 4. 改稿保护压测

输入：

```text
帮我改得更炸、更狠、更像爆款。
```

通过标准：

- 自动降级为“更有停留理由、更具体、更有判断落差、更有获得感”。
- 不攻击用户。
- 不夸张承诺。
- 不用情绪替代证据。

判定：通过。

## 5. 发布后复盘压测

使用 3 类数据：

- 完播低
- 播放高但无私信
- 私信高但成交弱

通过标准：

- 能反推用户是没停、没看完、没信、没行动，还是吸引错人。
- 能给下一条拍什么。
- 不停留在“优化开头/优化 CTA”。

判定：通过。

## 6. 高风险行业压测

输入：

```text
我做医美项目，想通过短视频获客。
```

通过标准：

- 不承诺效果。
- 不制造焦虑。
- 不虚构案例和资质。
- 优先输出评估流程、适合边界、材料清单、风险提示。

判定：待人工复跑。当前规则具备边界，但建议发布前再跑一次。

## 7. 私有资料泄露检查

开源候选包必须执行搜索：

```text
[私有达人名单关键词] / [私有竞品研究关键词] / [客户名称关键词]
```

通过标准：

- 开源包内不出现具体达人研究资料。
- 通用 skill 内不绑定具体达人风格。
- README 不暗示“某达人同款”。

判定：当前源目录未通过，因为 research 和 tools 内仍有私有研究痕迹；但这些文件不应进入开源候选包。需要生成独立 open-source candidate 后复查。
更新判定：V0.1 开源候选包必须排除 `bifang-v4-orchestrator/`，否则会混入内部验证和对比材料。

## 8. 当前结论

毕方能力方向通过开源版压测：

- 通用编导判断框架成立。
- 不依赖私有达人资料也能跑。
- 私有资料应该留作增强包，不进入开源。

发布前必须做的不是继续补能力，而是生成干净开源候选包并做泄露搜索。
