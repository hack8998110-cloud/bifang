# 毕方完整链路编导思维压测记录

> 日期：2026-07-19  
> 链路：`starter -> topic -> script -> review -> rewrite`  
> 目标：验证用户判断题、判断标准、信任证据、成交理由和承接动作，经过选题、脚本、审核、改稿后是否仍然保留。  
> 结论：链路基本成立；最大风险出现在 `rewrite` 阶段，当用户要求“更炸/更狠”时，可能牺牲证据和判断标准，需要加硬规则。

## 验收标准

一条完整链路通过，必须满足：

- `starter` 能把行业翻译成用户判断题。
- `topic` 能说明为什么先拍，以及需要什么证据。
- `script` 能把判断题、证据和成交理由写进正文。
- `review` 能打回没有编导思维的稿子。
- `rewrite` 改开头和节奏时，不改丢五个锚点。

## Case 1：系统门窗获客

### 1. starter 输出锚点

输入：

```text
我做系统门窗，想获客。
```

核心判断：

- 用户判断题：装修业主怎么判断系统门窗别买错，哪些细节影响隔音、保温、漏风和售后。
- 决策阶段：正在比较，且不信任商家的参数和套餐。
- 第一条先拍：买系统门窗别先问价格，先看这 3 个验收细节。
- 证据：型材截面、五金、密封条、安装缝隙、工地现场、售后边界。

### 2. topic 输出

```markdown
1. 买系统门窗别先问价格，先看这 3 个验收细节
   目标用户：准备装修或换窗的业主
   用户决策阶段：正在比较
   用户判断题：怎么判断这家门窗不是只会报低价，后期不会漏风、隔音差、售后扯皮
   用户为什么会看：买错代价高，且用户容易被参数和套餐绕晕
   用户看完拿走什么：验收时先看型材截面、五金、密封、安装缝隙
   成交理由：商家先教判断标准，用户更愿意发户型和窗户问题咨询
   可信证据：现场材料、安装过程、验收细节，不能虚构隔音数值
   脚本模型：教知识 + 晒过程
   推荐理由：先解决最卡的比较问题，再承接精准咨询
```

判定：通过。能看出“行业 -> 用户判断题 -> 选题”。

### 3. script 输出节选

```markdown
**开场第一句**：买系统门窗，你先别问多少钱一平，先看这 3 个地方，不然后面漏风、吵、售后扯皮，价格再低也没用。

0-3 s：拿一段型材截面和安装现场开场，先说“别先问价格”。
3-15 s：点出业主最怕的 3 个结果：漏风、隔音差、售后扯皮。
15-35 s：
第一，看型材截面和五金，不是只听名字。
第二，看密封条和接口，不是只看展厅样品。
第三，看安装缝隙和打胶细节，不然后期问题大多出在落地。
35-50 s：这几个点不是为了显得专业，是为了让你验收时知道该盯哪里。没有现场图、没有安装过程、只给你套餐报价的，先别急着定。
50-60 s：门窗买的是长期体验，不是当场便宜。你把户型和噪音问题发来，我先帮你判断该重点看哪一项。

**改稿锚点**：
- 用户问题：怕花高价买到只会包装概念的门窗。
- 判断标准：型材、五金、密封、安装、售后边界。
- 信任证据：现场材料和验收过程，不编隔音数值。
- 成交理由：用户拿到判断标准后，更愿意发户型进一步咨询。
- 承接动作：把你家窗户尺寸和噪音问题发来，先判断该不该换。
```

判定：通过。脚本没有只讲卖点，也没有虚构数据。

### 4. review 结果

```markdown
判定：通过

用户观看闸门：通过。用户判断题明确，是“怎么判断别买错”，不是商家自我介绍。
编导思维闸门：通过。先拍理由、证据承接、改稿锚点齐全。
商业审核：通过。成交理由自然，承接动作是低压咨询，不是硬广。
风险：如果用户后续补充真实案例，不能把“隔音提升多少”写成未验证事实。
```

判定：通过。

### 5. rewrite 风险测试

用户要求：

```text
帮我改得更狠一点，更有冲击力。
```

合格改稿方向：

```markdown
买系统门窗，最坑人的不是贵，是你只问了价格。
价格便宜，当天很爽；装完漏风、吵、售后扯皮，后面几年都难受。

你先看 3 个地方。
第一，看型材截面和五金，不要只听系列名字。
第二，看密封条和接口，展厅样品好看不代表你家落地也稳。
第三，看安装缝隙和打胶，门窗很多问题不是产品坏，是安装没盯住。

所以验收别只问“多少钱一平”，先问“这几个位置我能不能看现场”。
门窗买的是长期体验，不是当场便宜。你把户型和噪音问题发来，我先帮你判断该重点看哪一项。
```

判定：通过，但需加规则。  
原因：改得更有冲击可以，但必须保留型材、五金、密封、安装这些判断标准和“现场证据”，不能只剩情绪化开头。

## Case 2：职场沟通课卖课

### 1. starter 输出锚点

输入：

```text
我是职场沟通课老师，想卖课。
```

核心判断：

- 用户判断题：这个方法能不能解决我明天就要遇到的汇报、争取资源、被质疑场景。
- 决策阶段：没意识到问题到正在比较之间。
- 第一条先拍：汇报被老板打断，不是你表达差，是开头顺序错了。
- 证据：真实汇报场景复盘、前后两种表达对比。

### 2. topic 输出

```markdown
1. 汇报被老板打断，不是你表达差，是开头顺序错了
   目标用户：经常汇报被打断、方案推不动的职场人
   用户决策阶段：没意识到问题
   用户判断题：我汇报失败到底是内容不行，还是表达顺序错了
   用户为什么会看：场景具体，很多人当天或明天就会遇到
   用户看完拿走什么：汇报开头先讲结论、风险、需要老板拍板的点
   成交理由：用户能马上试一次，才愿意相信课程有用
   可信证据：前后表达对比、学员打码反馈、真实汇报文本
   脚本模型：教知识 + 做对比
   推荐理由：先给一个能立刻验证的小结果，再承接体验课或咨询
```

判定：通过。

### 3. script 输出节选

```markdown
**开场第一句**：汇报一开口就被老板打断，不一定是你方案差，很多时候是你第一句话顺序错了。

0-3 s：直接还原场景：“老板，我先讲一下背景……”然后被打断。
3-15 s：解释用户痛点：你以为讲完整才专业，老板听到第三句已经不知道你要他决定什么。
15-35 s：
别这样开头：“这个项目我们调研了很多资料……”
改成这样：“这个项目我建议先做 A，不建议做 B，风险是预算，今天需要您拍板的是上线时间。”
35-50 s：判断标准：汇报开头必须有结论、风险、需要对方决定的事。没有这 3 个，再详细也容易被打断。
50-60 s：你不是不会表达，你是让对方太晚知道他该干什么。把你的汇报开头发来，我先帮你看第一句话顺序。
```

判定：通过。能给具体场景和判断标准，没有承诺升职加薪。

### 4. review 结果

```markdown
判定：通过

用户观看闸门：通过。用户能认领“汇报被打断”的具体场景。
编导思维闸门：通过。用户判断题、判断标准、证据承接、承接动作齐全。
商业审核：通过。卖课承接较轻，没有硬广；但需要提醒不要承诺升职加薪。
```

判定：通过。

### 5. rewrite 风险测试

用户要求：

```text
改得更像爆款，更刺激一点。
```

不合格风险版本：

```markdown
你汇报总被打断，就是因为你太不会说话了。
真正厉害的人，一句话就能让老板听下去。
想升职，先学会汇报。
```

打回原因：

- 用户判断题被改丢：从“汇报开头顺序错了”变成泛泛攻击用户。
- 判断标准被改丢：没有结论、风险、拍板事项。
- 高风险承诺：暗示“想升职就学课”。
- 信任证据缺失：没有前后表达对比。

合格改稿方向：

```markdown
汇报被老板打断，别先怪老板没耐心。
很多时候，是你第一句话让他听不出要拍板什么。

你别这样开头：
“这个项目我们调研了很多资料……”

改成这样：
“这个项目我建议先做 A，不建议做 B。风险是预算，今天需要您拍板的是上线时间。”

记住一个判断标准：汇报开头先放结论、风险、拍板事项。
老板不是不想听细节，是他得先知道自己要决定什么。

把你的汇报第一句话发来，我先帮你看顺序。
```

判定：通过，但需加规则。  
原因：改稿目标含“爆款/刺激”时，必须先降级为“更有停留理由”，不能默认往攻击、夸张承诺和情绪刺激走。

## 总体结论

完整链路能跑通：

- `starter` 能判断用户真正想看什么。
- `topic` 能把判断题变成值得先拍的选题。
- `script` 能把判断标准和信任证据写进正文。
- `review` 能识别没有编导思维的脚本。
- `rewrite` 基础锚点存在，但需要增加“强刺激改稿保护”。

## 必须补的规则

1. `rewrite` 遇到“更炸/更狠/更刺激/更像爆款”时，不能牺牲用户判断题、证据和边界。
2. `rewrite` 改稿前必须先声明五个锚点是否完整；缺锚点时先补假设或要求用户补素材。
3. `review` 已能打回模型套壳，但要继续保持“编导思维闸门”优先于 7 维评分。
